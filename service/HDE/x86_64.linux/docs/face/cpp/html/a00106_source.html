1 QoS Provider {#DCPS_QoS_Provider}
2 ===========
3 
4 [TOC]
5 
6 This page provides information on using the QoS provider API of OpenSpliceDDS. The QoS provider API allows users to specify the QoS settings of their DCPS entities outside of application code in XML. This can be seen as a useful feature where code recompilation is restricted during the later stages of application development / during application support. The following sections explain the API in a language independent manner and explain how to build QoS Profiles in XML.

12 Introduction {#DCPS_QoS_Provider_Introduction}
13 -----------
14 
15 The QosProvider is delivered as part of the DCPS API of OpenSpliceDDS and has no factory. It is created as an object directly by the natural means in each language binding.
16 This is because it is not necessarily associated with a single DomainParticipant. Below you'll find a link to the available operations, which are explained in detail there.

30 XML file syntax {#DCPS_QoS_Provider_XMLsyn}
31 =============
32 
33 The XML configuration file must follow these syntax rules:
34 
35 - The syntax is XML and the character encoding is UTF-8.
36 - Opening tags are enclosed in `<>`; closing tags are enclosed in `</>`.
37 - A value is a UTF-8 encoded string. Legal values are alphanumeric characters.
38 - All values are case-sensitive unless otherwise stated.
39 - Comments are enclosed as follows: <\!-- comment -->.
40 - The root tag of the configuration file must be and end with .
41 
42 Please also see: [OMG DDS4CCM](http://www.omg.org/spec/dds4ccm/1.1/) specification for more details on the exact notation.
43 
44 The primitive types for tag values are specified in the following table:
45 
46 |Type | Format | Notes|
47 |-----|--------|------|
48 Boolean | yes, 1, true or BOOLEAN_TRUE | Not case-sensitive |
49 | | no, 0, false or BOOLEAN_FALSE | |
50 |Enum | A string. Legal values are the ones defined for QoS Policies in the DCPS IDL of DDS specification. | Must be specified as a string. (Do not use numeric values.)|
51 |Long | -2147483648 to 2147483647 or 0x80000000 to 0x7fffffff or LENGTH_UNLIMITED | A 32-bit signed integer|
52 |UnsignedLong | 0 to 4294967296 or 0 to 0xffffffff | A 32-bit unsigned integer|

54 Entity QoS {#DCPS_QoS_Provider_Entity_QoS}
55 ----------
56 
57 To configure the QoS for a DDS Entity using XML, the following tags have to be used:
58 - `<domainparticipant_qos>`
59 - `<publisher_qos>`
60 - `<subscriber_qos>`
61 - `<topic_qos>`
62 - `<datawriter_qos>`
63 - `<datareader_qos>`
64 
65 Each XML tag with an associated name can be uniquely identified by its fully qualified name in C++ style.
66 
67 ### QoS Policies
68 
69 The fields in a QosPolicy are described in XML using a 1-to-1 mapping with the equivalent IDL representation in the DDS specification. For example, the Reliability QosPolicy is represented with the following structures:
70 ~~~~~~~~~~~~~{.c}
71 struct Duration_t {
72  long sec;
73  unsigned long nanosec;
74 };
75 
76 struct ReliabilityQosPolicy {
77  ReliabilityQosPolicyKind kind;
78  Duration_t max_blocking_time;
79 };
80 ~~~~~~~~~~~~~
81 The equivalent representation in XML is as follows:
82 ~~~~~~~~~~~~~{.xml}
83 <reliability>
84  <kind></kind>
85  <max_blocking_time>
86  <sec></sec>
87  <nanosec></nanosec>
88  </max_blocking_time>
89 </reliability>
90 ~~~~~~~~~~~~~
91 
92 ### Sequences
93 
94 In general, the sequences contained in the QoS policies are described with the following XML format:
95 ~~~~~~~~~~~~~{.xml}
96 <a_sequence_member_name>
97  <element>...</element>
98  <element>...</element>
99  ...
100 </a_sequence_member_name>
101 ~~~~~~~~~~~~~
102 Each element of the sequence is enclosed in an `<element>` tag., as shown in the following example:
103 ~~~~~~~~~~~~~{.xml}
104 <property>
105  <value>
106  <element>
107  <name>my name</name>
108  <value>my value</value>
109  </element>
110  <element>
111  <name>my name2</name>
112  <value>my value2</value>
113  </element>
114  </value>
115 </property>
116 ~~~~~~~~~~~~~
117 
118 A sequence without elements represents a sequence of length 0. For example: `<a_sequence_member_name/>` As a special case, sequences of octets are represented with a single XML tag enclosing a sequence of decimal/hexadecimal values between 0..255 separated with commas. For example:
119 ~~~~~~~~~~~~~{.xml}
120 <user_data>
121  <value>100,200,0,0,0,223</value>
122 </user_data>
123 <topic_data>
124  <value>0xff,0x00,0x8e,0xEE,0x78</value>
125 </topic_data>
126 ~~~~~~~~~~~~~
127 ### Arrays
128 
129 In general, the arrays contained in the QoS policies are described with the following XML format:
130 ~~~~~~~~~~~~~{.xml}
131 <an_array_member_name>
132  <element>...</element>
133  <element>...</element>
134  ...
135 </an_array_member_name>
136 ~~~~~~~~~~~~~
137 Each element of the array is enclosed in an `<element>` tag. As a special case, arrays of octets are represented with a single XML tag enclosing an array of decimal/hexadecimal values between 0..255 separated with commas. For example:
138 ~~~~~~~~~~~~~{.xml}
139 <datareader_qos>
140  ...
141  <user_data>
142  <value>100,200,0,0,0,223</value>
143  </user_data>
144 </datareader_qos>
145 ~~~~~~~~~~~~~
146 ### Enumerations
147 
148 Enumeration values are represented using their IDL string representation. For example:
149 ~~~~~~~~~~~~~{.xml}
150 <history>
151  <kind>KEEP_ALL_HISTORY_QOS</kind>
152 </history>
153 ~~~~~~~~~~~~~
154 
155 ### Time values (Durations)
156 
157 Following values can be used for fields that require seconds or nanoseconds:
158 - DURATION_INFINITE_SEC
159 - DURATION_ZERO_SEC
160 - DURATION_INFINITE_NSEC
161 - DURATION_ZERO_NSEC
162 
163 The following example shows the use of time values:
164 ~~~~~~~~~~~~~{.xml}
165 <deadline>
166  <period>
167  <sec>DURATION_INFINITE_SEC</sec>
168  <nanosec>DURATION_INFINITE_NSEC</nanosec>
169  </period>
170 </deadline>
171 ~~~~~~~~~~~~~

173 QoS Profiles {#DCPS_QoS_Provider_QoS_Profiles}
174 ------------
175 
176 A QoS profile groups a set of related QoS, usually one per entity. For example:
177 ~~~~~~~~~~~~~{.xml}
178 <qos_profile name="StrictReliableCommunicationProfile">
179  <datawriter_qos>
180  <history>
181  <kind>KEEP_ALL_HISTORY_QOS</kind>
182  </history>
183  <reliability>
184  <kind>RELIABLE_RELIABILITY_QOS</kind>
185  </reliability>
186  </datawriter_qos>
187  <datareader_qos>
188  <history>
189  <kind>KEEP_ALL_HISTORY_QOS</kind>
190  </history>
191  <reliability>
192  <kind>RELIABLE_RELIABILITY_QOS</kind>
193  </reliability>
194  </datareader_qos>
195 </qos_profile>
196 ~~~~~~~~~~~~~
197 QoS Profiles with a Single QoS
198 
199 The definition of an individual QoS is a shortcut for defining a QoS profile with a single QoS. For example:
200 ~~~~~~~~~~~~~{.xml}
201 <datawriter_qos name="KeepAllWriter">
202  <history>
203  <kind>KEEP_ALL_HISTORY_QOS</kind>
204  </history>
205 </datawriter_qos>
206 ~~~~~~~~~~~~~
207 is equivalent to the following:
208 ~~~~~~~~~~~~~{.xml}
209 <qos_profile name="KeepAllWriter">
210  <datawriter_qos >
211  <history>
212  <kind>KEEP_ALL_HISTORY_QOS</kind>
213  </history>
214  </datawriter_qos >
215 </qos_profile>
216 ~~~~~~~~~~~~~

218 Example {#DCPS_QoS_Provider_Example}
219 =======
220 consider the following XML file that describes two QoS profiles;
221 - FooQosProfile
222  + DataReaderQos - TransientKeepLast
223  + DataWriterQos - Transient
224 - BarQosProfile
225  + DataWriterQos - Persistent
226 ~~~~~~~~~~~~~{.xml}
227 <dds xmlns="http://www.omg.org/dds/"
228  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
229  xsi:schemaLocation="file://DDS_QoSProfile.xsd">
230  <qos_profile name="FooQosProfile">
231  <datareader_qos name="TransientKeepLast">
232  <durability>
233  <kind>TRANSIENT_DURABILITY_QOS</kind>
234  </durability>
235  <history>
236  <kind>KEEP_LAST_HISTORY_QOS</kind>
237  <depth>5</depth>
238  </history>
239  </datareader_qos>
240  <datawriter_qos name="Transient">
241  <durability>
242  <kind>TRANSIENT_DURABILITY_QOS</kind>
243  </durability>
244  </datawriter_qos>
245  </qos_profile>
246  <qos_profile name="BarQosProfile">
247  <datawriter_qos name="Persistent">
248  <durability>
249  <kind>PERSISTENT_DURABILITY_QOS</kind>
250  </durability>
251  </datawriter_qos>
252  </qos_profile>
253 </dds>
254 ~~~~~~~~~~~~~

305 Known limitations {#DCPS_QoS_Provider_Limitations}
306 =================
307 - Inheritance of QoS policies and QoS profiles in XML using the "base_name" attribute is not supported yet.
308 - The "topic_filter" attribute for writer,reader, and topic QoSs to associate a set of topics to a specific QoS when that QoS is part of a DDS profile, is not supported yet.
309 - All leading and trailing spaces should be removed from the string before it is processed. For example, "<tag> value </tag>" should be the same as "<tag>value</tag>"
310 - Currently parsing of sequences and arrays only supports sequence of strings
