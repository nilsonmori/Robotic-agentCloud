95 
96 Those statuses may be classified in:
97 
98 - Read communication statuses: i.e., those that are related to arrival of data, namely DATA_ON_READERS and DATA_AVAILABLE.
99 - Plain communication statuses: i.e., all the others. Read communication statuses are treated slightly differently than the others for they don’t change independently.
100  In other words, at least two changes will appear at the same time (DATA_ON_READERS + DATA_AVAILABLE) and even several of the last kind may be part of the set. This ‘grouping’ has to be communicated to the application.
101 
102 Changes in Status
103 -----------------
104 Associated with each one of an Entity’s communication status is a logical StatusChangedFlag. This flag indicates
105 whether that particular communication status has changed since the last time the status was ‘read’ by the application. The
106 way the status changes is slightly different for the Plain Communication Status and the Read Communication status.
107 
108 ![StatusChangedFlag indicates if status has changed] (@ref InfrastructureModule_Status_CIS_UML.png)
109 
110 Changes in Plain Communication Status
111 -------------------------------------
112 For the plain communication status, the StatusChangedFlag flag is initially set to FALSE. It becomes TRUE whenever the plain communication status changes and it is reset to FALSE each time the application accesses the plain communication status via the proper get_<plain communication status> operation on the Entity.
113 
114 ![Changes in StatusChangedFlag for plain communication status] (@ref InfrastructureModule_Status_CIPS_UML.png)
115 
116 The communication status is also reset to FALSE whenever the associated listener operation is called as the listener implicitly accesses the status which is passed as a parameter to the operation.
117 The fact that the status is reset prior to calling the listener means that if the application calls the get_<plain communication status> from inside the listener it will see the status already reset.
118 
119 An exception to this rule is when the associated listener is the 'nil' listener.
120 The 'nil' listener is treated as a NO-OP and the act of calling the 'nil' listener does not reset the communication status.
121 
122 For example, the value of the StatusChangedFlag associated with the REQUESTED_DEADLINE_MISSED status will become TRUE each time new deadline occurs (which increases the total_count field within RequestedDeadlineMissedStatus). The value changes to FALSE when the application accesses the status via the corresponding get_requested_deadline_missed_status method on the proper Entity.
123 
124 
125 Changes in Read Communication Status
126 --------------------------------------
127 For the read communication status, the StatusChangedFlag flag is initially set to FALSE.
128 The StatusChangedFlag becomes TRUE when either a data-sample arrives or else the ViewState, SampleState, or InstanceState of any existing sample changes for any reason other than a call to DataReader::read, DataReader::take or their variants.
129 Specifically any of the following events will cause the StatusChangedFlag to become TRUE:
130 
131 - The arrival of new data.
132 - A change in the InstanceState of a contained instance. This can be caused by either:
133  - The arrival of the notification that an instance has been disposed by:
134  - the DataWriter that owns it if OWNERSHIP QoS kind=EXCLUSIVE
135  - or by any DataWriter if OWNERSHIP QoS kind=SHARED.
136  - The loss of liveliness of the DataWriter of an instance for which there is no other DataWriter.
137  - The arrival of the notification that an instance has been unregistered by the only DataWriter that is known to be writing the instance.
138 
139 
140 Depending on the kind of StatusChangedFlag, the flag transitions to FALSE again as follows:
141  - The DATA_AVAILABLE StatusChangedFlag becomes FALSE when either the corresponding listener operation (on_data_available) is called or the read or take operation (or their variants) is called on the associated DataReader.
142  - The DATA_ON_READERS StatusChangedFlag becomes FALSE when any of the following events occurs:
143  - The corresponding listener operation (on_data_on_readers) is called.
144  - The on_data_available listener operation is called on any DataReader belonging to the Subscriber.
145  - The read or take operation (or their variants) is called on any DataReader belonging to the Subscriber.
